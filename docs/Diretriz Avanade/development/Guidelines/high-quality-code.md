# High Quality

For quality this means not only to stick to the processes but also to always refer to your supervisor and the technical lead in case some team member notices one of the following things:

- The existing code is not optimal regarding algorithms, data structures, configuration settings etc. The technical lead should be notified about this, and he will add this as another improvement
  suggestion to the existing list.

- There are gaps in the processes which should be closed (e.g. it would be valuable to have one more field/information in a work item).

- Some important aspect of the application (old or new) is not documented well or documented at all.

- Some reoccurring work is done manually and should be automated or improved, especially in the configuration and deployment area.

- Some configuration or automation function is missing in the development environment which would increase productivity.

- There is room of improvement in existing processes which would speed up and/or increase quality of development and testing work.
