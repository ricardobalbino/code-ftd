3.4 Decisão Técnica — Power Pages para o Simulador Comercial

Apresentação por Fernando Mauricio da Costa Filho:

Premissas do simulador:
•	Liberdade de layout maior que o Dynamics nativo.
•	Interface amigável com performance independente do número de produtos.
•	Incluído na licença existente (sem custo adicional).
•	Manutenção prática.
•	Suporte a mobile e desktop.

3 opções avaliadas:
1.	Power Pages (escolhida): portal apartado, responsivo, single page por enquanto. Autenticação via Microsoft Entra ID (usuários internos com licença Enterprise) — sem custo adicional de licenciamento (confirmado com a Microsoft por e-mail).
2.	Custom Page (HTML/CSS dentro do Dynamics): descartada — experiência mobile limitada pelo app nativo do Dynamics for Sales.
3.	Canvas App: descartada — necessitaria manter aplicativo desktop e mobile separados, gerando manutenção em 2 lugares para alterações de front-end.

Estado atual do desenvolvimento:
•	Single page de adição de produto individual em desenvolvimento.
•	Ainda sem home page, header, footer ou contexto geral de portal.
•	Duas opções de entrega sendo consideradas: (1) redirecionamento via botão no Dynamics ou (2) embed dentro do Dynamics.
•	Intenção final: portal apartado independente.
•	Apresentação completa da estrutura do Power Pages planejada para após a entrega interna (sexta-feira), com sessão para Avanade na semana seguinte.

Estratégia de performance:
•	Mínimo de consultas ao Dataverse — dados carregados em cache no front-end.
•	Filtros dinâmicos processados no front-end (sem fetch ao Dataverse a cada interação).
•	Cálculos em tempo real via JavaScript no front-end.
•	Validação no back-end (Azure Functions) como segunda camada de segurança — garante integridade mesmo se o usuário não usar o simulador.
•	Uso de PowerFX para cálculos com valores totais (substituindo valores acumulados entre produtos, que causavam lentidão).

Cálculos — arquitetura atual:
•	Cálculo de produtos da proposta: se ≤50 produtos, processado no Dataverse (plugin síncrono dentro de 2 minutos). Se >50 produtos, enviado para Azure Function.
•	Cálculo da proposta comercial: feito por Azure Function.
•	Power Pages reutilizará as Azure Functions existentes, com refatoração/revisão planejada.

3.5 Estrutura de Código

•	Plugins: separados por entidade (contas, etc.). Índice de bugs muito baixo após 1,5 ano de reestruturação pela Squad (2 bugs no mês anterior).
•	JavaScript: formulários passaram por refatoração (assessment anterior). Algumas partes já no novo padrão, outras ainda como legado.
•	Azure Functions: renovação automática e cálculos de proposta comercial.
•	Batch projects: para atualizações em massa — executados em VM solicitada ao time de infra (não na máquina do desenvolvedor).
•	Actions: existem actions no Dynamics que disparam Power Automates. Não há actions expostas para consumo externo — sistemas externos consultam via OData ou FetchXML.

•	Azure Key Vault: usado para Azure Functions (tokens, URLs), mas não dentro dos Power Automates (usam variáveis de ambiente para URLs e dados).
•	Conexões existentes para child flows: a verificar — atenção ao limite de conexões por usuário de serviço.
•	Tabelas de suporte:

3.11 Governança e Padrões

•	Prefixo de tabelas/campos: "ftd_" (publicador padrão FTD).
•	Não há convenção de nomenclatura além do prefixo — prática é deixar nomes de variáveis/campos claros e descritivos.
•	Security roles: problema reconhecido — ambientes não equalizados, muitos perfis possivelmente duplicados ou que poderiam ser unificados. Equalização no roadmap mas ainda não executada.
•	Processo de bugs: N1 feito por equipe de sustentação separada (portal à parte). Se não resolvido, bug é criado no DevOps e atribuído ao time do CRM. Story bugs são criados diretamente no DevOps associados à user story.
