---

## 📋 Objetivo

Checklist mestre para Product Owners. Consolida todas as responsabilidades, validações e melhores práticas para garantir excelência na gestão de produto.

---

## 📊 Áreas de Responsabilidade

### 1. Vision & Strategy

- [ ] **Product Vision documentada e comunicada**
  - Vision statement claro
  - Alinhado com estratégia de negócio
  - Compreendido pelo time

- [ ] **Roadmap atualizado**
  - Próximos 3-6 meses visível
  - Prioridades claras
  - Milestones definidos

- [ ] **Success Metrics definidas**
  - KPIs mensuráveis
  - Baseline estabelecido
  - Tracking configurado

---

### 2. Backlog Management

#### 2.1 Backlog Health

- [ ] **Backlog priorizado**
  - Top 10 stories ready for sprint
  - Priorização justificada
  - Stakeholders alinhados

- [ ] **Stories bem escritas**
  - Formato user story consistente
  - Acceptance criteria claros
  - INVEST criteria atendidos

- [ ] **Backlog groomed regularmente**
  - Refinement sessions agendadas
  - Stories estimadas
  - Dependencies mapeadas

#### 2.2 Backlog Hygiene

- [ ] **Items obsoletos removidos**
  - Review trimestral
  - Arquivar não-prioridades
  - Atualizar estimates antigos

- [ ] **Epics bem definidos**
  - Objetivo claro
  - Scope delimitado
  - Stories derivadas

---

### 3. Sprint Ceremonies

#### 3.1 Sprint Planning

- [ ] **Sprint Goal definido**
  - Objetivo único e claro
  - Alinhado com roadmap
  - Comunicado ao time

- [ ] **Stories selecionadas atendem DoR**
  - Definition of Ready checklist
  - Sem blockers conhecidos
  - Acceptance criteria aprovados

- [ ] **Capacity respeitada**
  - Não over-commit
  - Buffer para imprevistos
  - Holidays considerados

#### 3.2 Daily Participation

- [ ] **Disponível para esclarecer dúvidas**
  - Resposta em < 4 horas
  - Decisões rápidas
  - Não bloquear time

#### 3.3 Sprint Review

- [ ] **Demo preparada**
  - Stories done showcased
  - Stakeholders convidados
  - Feedback coletado

- [ ] **Acceptance realizado**
  - Stories validadas contra AC
  - PO sign-off dado
  - Issues documentados

#### 3.4 Retrospective

- [ ] **Participação ativa**
  - Perspectiva de produto trazida
  - Ações de melhoria acordadas
  - Follow-up de ações anteriores

---

### 4. Stakeholder Management

- [ ] **Stakeholders identificados**
  - Mapa de stakeholders
  - Interesses conhecidos
  - Frequência de comunicação definida

- [ ] **Expectativas gerenciadas**
  - Scope claro
  - Timeline comunicado
  - Riscos compartilhados

- [ ] **Feedback loop estabelecido**
  - Canais de feedback claros
  - Feedback incorporado em backlog
  - Stakeholders informados de decisões

---

### 5. Product Quality

- [ ] **Definition of Done respeitado**
  - DoD aplicado a todas stories
  - Exceções documentadas
  - Qualidade não comprometida

- [ ] **Acceptance Testing executado**
  - UAT em staging
  - Edge cases verificados
  - UX/UI conforme esperado

- [ ] **Release criteria atendidos**
  - Feature complete
  - Bugs críticos resolvidos
  - Documentação pronta

---

### 6. Metrics & Reporting

- [ ] **Velocity tracked**
  - Histórico de sprints
  - Tendências identificadas
  - Forecast atualizado

- [ ] **Product metrics monitoradas**
  - Usage analytics
  - User feedback
  - Business KPIs

- [ ] **Reporting para stakeholders**
  - Status reports regulares
  - Burndown/burnup charts
  - Roadmap updates

---

## 📅 Cadência de Atividades

### Diário
- [ ] Revisar blockers do time
- [ ] Responder dúvidas pendentes
- [ ] Check de métricas de produto

### Semanal
- [ ] Backlog refinement (1-2h)
- [ ] Stakeholder touchpoints
- [ ] Review de feedback de usuários

### Por Sprint
- [ ] Sprint Planning preparation
- [ ] Sprint Review execution
- [ ] Retrospective participation
- [ ] Backlog reprioritization

### Mensal
- [ ] Roadmap review e update
- [ ] Stakeholder presentation
- [ ] Competitor analysis

### Trimestral
- [ ] Strategic alignment check
- [ ] Backlog cleanup
- [ ] Metrics deep dive

---

## 🚨 Red Flags - Quando Agir

| Situação | Ação |
|----------|------|
| Sprint sem stories ready | Emergency refinement |
| Stakeholder bypass | Realinhamento imediato |
| Scope creep | Renegociar ou criar nova story |
| Time ocioso esperando decisão | Priorizar resposta |
| Velocity caindo | Investigar com SM |
| Feedback negativo consistente | Pivot ou deep dive com usuários |

## 📝 Templates Rápidos

### Sprint Goal
```
Neste sprint, vamos [OBJETIVO] para que [BENEFÍCIO], 
o que será demonstrado por [EVIDÊNCIA DE SUCESSO].
```

### Acceptance Criteria
```
Dado [CONTEXTO],
Quando [AÇÃO],
Então [RESULTADO ESPERADO].
```

### Priority Justification
```
Esta story é [PRIORITY] porque:
- Business value: [ALTO/MÉDIO/BAIXO]
- User impact: [DESCRIÇÃO]
- Urgência: [RAZÃO]
- Alternativas consideradas: [LISTA]
```
