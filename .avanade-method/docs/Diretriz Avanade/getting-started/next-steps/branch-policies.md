# Branch Policies

As of BCA version 3.0, branch policies are not automatically configured during the setup which means that this should be done as required by the project needs. Branch policies are an essential part of quality assurance to only allow certain users to allow commits to the main branch. 

| Policy                                                              | Sub-Policy                  | Recommended Setting                                                                                                                                                              | Comments                                                                                                                                                                                                                                                                                  |
| ------------------------------------------------------------------- | --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Require a minimum number of reviewers                               | Minimum number of reviewers | 1                                                                                                                                                                                | At least one reviewer must approve but based on the project setup and eventually more should be added. This should also be considered in combination with the “Automatically include code reviewers” policy.                                                                              |
| Allow users to approve their own changes.                           | Depends                     | In projects where code is already productive, this should but off. But it be allowed for example in an early project stage or in projects where automated tests are very mature. |
| Allow completion even if some reviewers vote “Waiting” or “Reject”. | No                          |                                                                                                                                                                                  |
| Reset code reviewer votes when there are new changes.               | Depends                     |                                                                                                                                                                                  |
| Check for linked work items                                         | Policy requirement          | Required                                                                                                                                                                         | All changes must always be traceable back to the related work item(s).                                                                                                                                                                                                                    |
| Check for comment resolution                                        | Policy requirement          | Required                                                                                                                                                                         |                                                                                                                                                                                                                                                                                           |
| Limit merge types                                                   | Allowed merge types         | Depends                                                                                                                                                                          | It is recommended to only allow one merge type to have a consistent merge history. Some projects prefer squash merge because it preserves are clear linear history without too much clutter.                                                                                              |
| Build validation                                                    | Build policy                | Required                                                                                                                                                                         | The regular guard build (4.4) should be executed, and a successful completion should be pre-requisite for a successful pull request. This guard build should also cover code analysis policies.                                                                                           |
| Require approval from additional services                           | Status policy               | To be defined                                                                                                                                                                    | This should be evaluated for applicability.                                                                                                                                                                                                                                               |
| Automatically include code reviewers                                | Policy requirement          | Depends                                                                                                                                                                          | In combination with the “Minimum number of reviewers” setting, it should be decided if approval from a certain group is always required (this can be also filtered by code paths), for example approval from technical architects when a certain part of the application code is touched. |
